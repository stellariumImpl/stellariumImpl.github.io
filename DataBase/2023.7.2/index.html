Summary of Storage

Recap

The storage module is responsible for data storage, retrieval and management on disk and memory, and provides fine-grained data operation interface to the upper layer. As this module is less coupled with other modules, we can visualize it as a dedicated database component, storage engine, and many databases currently support pluggable storage engines, such as MySQL supports InnoDB, MyISAM, Memory and other storage engines. Any design of memory, cache, read and write process is based on the storage structure, so the storage structure and storage engine features and performance are very close. Compared with the vast number of storage engines, there are few storage structures. B-tree, as a storage structure well suited for disk, has been dominating the storage field since the rise of relational data building, but in recent years, with the rise of distributed technology and the development of solid-state drives, another storage structure, LSM tree, has become increasingly popular.

Data Files & Index Files

The storage engine mainly needs to store data files and index files, and we need to convert the abstract individual data into concrete files.

There are three forms of data file organization:

  • Index organization table: data files and index files are combined together, directly storing data records inside the index file.
  • Heap organization table: The data file is an “unordered heap” data structure, in which data records are generally not required to have a specific order, although the heap organization table data file is unordered, but we can use the index structure in the index file to quickly get the location of the data in the heap, so as to carry out rapid access.
    Because data insertion is unordered, while index-organized tables need to insert data in the order of indexes, heap-organized tables have stronger write performance than index-organized tables and weaker read performance than index-organized tables.
  • Hash-organized table: the records are scattered in a bucket, and the hash value of each record’s primary key determines which bucket the record belongs to. The part that records the hash value is the index file, and the part that records the hash result is the data file.

In fact, the organization form of data file still has a smaller impact on performance, InnoDB storage engine uses indexed organization table, while Oracle and PostgreSQL are heap organization table, but we seldom distinguish their performance from this perspective, because they all use B-tree index, and index as our main entrance to read and write data, index file The organization form is actually more closely related to the performance of the storage engine, for example, if we change the organization form of the index file from B-tree to LSM-tree, then the difference in performance is much larger. So, we will index file organization form specifically called the storage engine’s storage structure.

Storage Structure

In-place update structure: B & B+ Tree, overwrite old records directly to store updates.

This in-place update structure will only store the latest version of each record, so read performance is better, but writes become more costly because updates cause random IO.

Out-of-place update structure: LSM Tree, store the updated content in the new location instead of overwriting the old one.

This update has obviously higher write performance because it is sequential, but read performance is sacrificed because we have to scan multiple locations to read the result we want. In addition this data structure requires a data integration process, mainly to prevent infinite expansion of data, and this process is the very critical Compaction process.

History of Storage Structure

1970

Professor Rudolf Bayer presented B Tree in “Organization and Maintenance of Large Ordered Indices”.

1996

LSM concept was born with the publication of “The Log-Structured Merge-Tree”.

2003

Google’s distributed file system was released, appending consistency is guaranteed to be stronger than rewriting, and since then distributed file systems have basically been more recommended for appending write files.

2014

SSD began to enter mass commercialization, LSM-Tree read performance on SSD was greatly improved (random reads reduced seek time), LSM-Tree was further developed as appending writes improved SSD lifetime. So LSM-Tree became more write-friendly on SSD.
